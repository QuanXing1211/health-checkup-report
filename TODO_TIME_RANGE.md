# 可选时间范围输入待办

## 背景

当前 `health-checkup-report` 的生成流程把 `--start` / `--end` 当成固定输入使用，但实际需求已经变成：

- 用户在企微里可以显式传起始和终止时间
- 如果用户没传时间，则由系统去某个默认时间接口获取时间范围
- 后续所有需要时间的接口，都应该使用最终解析出来的“有效时间范围”

这意味着时间逻辑不能继续散落在各个接口调用里，应该收敛到一个统一入口。

## 目标

1. 支持用户可选传入 `start` 和 `end`
2. 不传时走默认时间范围来源
3. 所有下游需要时间的调用都只依赖同一个“有效时间范围”
4. 文档和 skill 描述同步更新，避免仍然写成“start 必填”

## 建议改造顺序

### 1. 先更新 skill 描述

需要修改 [`SKILL.md`](./SKILL.md)：

- 目前写的是 `--start` 必填
- 要改成“时间可选”
- 说明用户传了就透传，不传就走默认时间范围

如果 README 也承担对外使用说明，同步更新 [`README.md`](./README.md)。

### 2. 在入口层统一解析时间

建议在 [`health_report.js`](./health_report.js) 增加一个统一的时间解析步骤，例如：

- 用户同时传了 `start/end`，直接使用
- 用户完全没传时间，调用默认时间范围获取逻辑
- 用户只传了一半，建议追问，不要猜

这一步产出一个最终对象，例如：

```js
{
  start: '2026-06-01',
  end: '2026-06-16'
}
```

后续所有逻辑都只依赖这个结果。

### 3. 让主流程使用有效时间范围

`health_report.js` 里以下位置都应改成使用统一后的时间范围：

- `exportConfiguredXdrTables(...)`
- `collectReportData(...)`
- `fetchAlertTableCount(...)`
- `fetchSecurityLogCount(...)`
- `fetchAlertReductionRate(...)`

目标是避免同一轮生成里，有的接口用用户输入，有的接口又去取默认值。

### 4. 保持底层时间校验纯粹

[`src/xdr_asset_client.js`](./src/xdr_asset_client.js) 里现在有 `resolveIncidentTimeRange(...)`。

建议它继续只负责：

- 校验时间格式
- 校验 `start <= end`

不要让它承担“默认时间来源”的职责。默认值应该放在入口层统一决定。

### 5. 单独封装默认时间来源

默认时间范围的获取逻辑建议抽成独立函数或服务，后续只需要替换这一处：

- 入口层调用它
- 它返回最终 `start/end`
- 下游完全不知道默认值是怎么来的

这样后续如果默认接口变了，只改一个地方。

## 受影响文件

- [`SKILL.md`](./SKILL.md)
- [`README.md`](./README.md)
- [`health_report.js`](./health_report.js)
- [`src/xdr_asset_client.js`](./src/xdr_asset_client.js)
- 可能还会影响调用时间参数的其他辅助函数，但原则是统一接收最终时间范围

## 验收标准

1. 用户传了 `start/end` 时，系统使用用户传入时间
2. 用户没传时间时，系统能从默认时间来源拿到时间范围
3. 后续所有需要时间的接口都使用同一个最终时间范围
4. `skill.md` 和 README 不再把 `--start` 写成绝对必填
5. 未传时间时不会因为底层校验提前报错
